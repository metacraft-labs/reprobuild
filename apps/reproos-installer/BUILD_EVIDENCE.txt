M9.R.19 closure (eli-wsl, 2026-06-23):
================================================

M9.R.19.1 -- qt6-quickcontrols2 from-source recipe
  Landed `recipes/packages/source/qt6-quickcontrols2/repro.nim`. Builds
  from the shared qtdeclarative-everywhere-src-6.8.1.tar.xz tarball
  (referenced via `file:../qt6-declarative/vendor/...` so the 36-MiB
  blob isn't duplicated). Upstream Qt merged QuickControls2 INTO
  qtdeclarative at Qt 6.2; the standalone qtquickcontrols2-everywhere-
  src-<ver>.tar.xz returns HTTP 404 from download.qt.io (verified
  M9.R.19).

M9.R.19.1.2 -- NIX_CFLAGS_COMPILE leak fix
  Cleared NIX_CFLAGS_COMPILE for the qt6-quickcontrols2 cmake configure
  via extraEnv. Without this fix, CMake's implicit-include-dir scan
  picks up sibling Qt include dirs from the nix-shell env and
  deduplicates the explicit -I.../QtCore + -isystem .../usr/include
  flags out of the per-target compile commands; the actual gcc
  invocation then lacks them and fails with
  "QtCore/qtsymbolmacros.h: No such file or directory" when compiling
  qrc_jsRootMetaTypes_init.cpp.

M9.R.19.3 -- ISO integration
  stage-de-rootfs.sh now consumes
  apps/reproos-installer/.repro/output/install/usr/bin/reproos-installer
  as the canonical source for the installer binary. The script exits
  with code 66 if the binary is missing (was a soft skip in M9.R.18.14).
  Defaults the SDDM autologin Session= to reproos-installer (was
  plasma in M9.R.18.1 as a placeholder). reproos-iso/repro.nim
  declares the installer binary path as an extraInput so the engine
  refingerprints the ISO when the wizard changes.

M9.R.19.1.3 -- qt6-quickcontrols2 shim
  The cmake_package-based build of the qtdeclarative tarball hits a
  deterministic cmake_autogen wedge in the
  qtquickcontrols2nativestyleplugin_autogen target after producing
  51 of 60 moc files. Reproduced across two independent eli-wsl runs.
  Switched the recipe to a shim that copies the QuickControls2
  artifacts out of the sibling qt6-declarative install mirror. The
  share-only fast-path in the from-source tool resolver matches as
  soon as Qt6QuickControls2Config.cmake lands at the expected layout.

M9.R.19.2 -- engine pipeline end-to-end
  The cmake_package convention's M9.R.15i.5 auto-thread-cmake-
  config-dirs walker assumes the recipe lives under
  recipes/packages/source/<x>/ so it finds sibling deps via
  parentDir(projectRoot)/<dep>. For apps/ recipes the sibling layout
  is wrong (apps/qt6-declarative doesn't exist), so no
  -DQt6Qml_DIR / -DQt6Quick_DIR is emitted and cmake configure
  hard-fails. Passed Qt6Qml_DIR, Qt6Quick_DIR, Qt6QuickControls2_DIR,
  + transitive component dirs explicitly via cacheVars.

  Engine-built artifact:
    /opt/repro/reprobuild/apps/reproos-installer/.repro/output/install/usr/bin/reproos-installer
    89,560 bytes, ELF executable.
  Smoke run:
    LD_LIBRARY_PATH=<from-source qt6/libxkbcommon/mesa> \
    QT_QPA_PLATFORM=offscreen reproos-installer --version
    => "ReproOS Installer 0.1.0"
