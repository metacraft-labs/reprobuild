# Nim

Nim is reprobuild's first-class language: the engine itself is written
in Nim, the `repro.nim` project DSL is Nim, and the Nim Mode 3 pilot
is the most exercised path through the standard provider.

## Modes available

- **Mode 3** (recommended for new projects): minimal `repro.nim`,
  scanner-driven dep graph, no `.nimble` needed.
- **Mode 2**: existing `<pkg>.nimble` is recognized and lifted into
  the action graph.
- **Mode 1**: layout-as-manifest (M48, 2026-05-29). Drop sources under
  `apps/<name>/src/<name>.nim` + `libs/<name>/src/<name>.nim`, run
  `repro build` — no project file needed. Fixture:
  `reprobuild-examples/mode1/nim-binary-with-library/`. See
  [The Three Modes §Mode 1](../three-modes.md#mode-1--layout-as-manifest).

## Quickstart (Mode 3)

Minimal `repro.nim`:

```nim
import repro_project_dsl

package greet:
  uses:
    "nim >=2.2 <3.0"
  library greet

package hello:
  uses:
    "nim >=2.2 <3.0"
  executable hello:
    discard

include "repro.scanned-deps.nim"
```

Minimal layout:

```text
my-nim-workspace/
  repro.nim
  repro.scanned-deps.nim     # generated by `repro deps refresh`
  src/
    hello.nim                # `import greet`
    greet.nim                # exports `proc greet*(...)`
```

Build:

```text
repro build
```

Outputs:

```text
.repro/build/greet/libgreet.a
.repro/build/hello/hello[.exe]
```

The Nim convention emits a three-phase build for each executable:
`nim c --compileOnly` to produce C, `gcc -c` per emitted file, then a
final `gcc -o` link picking up dependent archives.

## Source layout

The Nim convention recognizes two layouts:

**Layout A (single-package, flat).** All sources directly under `src/`:

```text
my-pkg/
  repro.nim
  src/
    main.nim
    helpers.nim
```

**Layout B (multi-package workspace).** One subdirectory per package,
each with its own `src/`:

```text
my-monorepo/
  repro.nim
  greet/src/greet.nim
  hello/src/hello.nim
```

The Mode 3 pilot fixture uses Layout A with both packages sharing the
`src/` directory (Nim's flat module space supports it). Real workspaces
typically prefer Layout B.

Reference fixture:
[`reprobuild-examples/nim/mode3-pilot/`](https://github.com/metacraft-labs/reprobuild-examples/tree/main/nim/mode3-pilot).

## Mode 2 escape hatch

If you have an existing `<pkg>.nimble` file, drop it next to your
`reprobuild.nim` and the standard provider will recognize the
package, read the `.nimble`'s `bin`, `srcDir`, `requires`, etc., and
build accordingly:

```text
my-pkg/
  reprobuild.nim
  mypkg.nimble                 # ecosystem manifest
  src/
    mypkg.nim
```

Use Mode 2 when:
- You're publishing the package to `nimble.directory` and need a
  canonical `.nimble`.
- You depend on third-party `nimble` packages that need
  `nimble install` resolution.
- Existing tooling (Nimble registry, `nimble test`, etc.) needs the
  manifest.

## Cross-language

Nim is supported in three cross-language combinations:

- [Nim ↔ C/C++](../cross-language/nim-and-c-cpp.md) — via `{.importc.}`
  / `{.exportc.}` pragmas, the most common case (Nim emits C).
- [Nim ↔ Rust](../cross-language/nim-and-rust.md) — via Rust
  `#[no_mangle] extern "C"` + Nim `{.importc.}`.

## The scanner

The Nim scanner (`repro_core/nim_dep_scanner`) recognizes:

- `import foo`
- `import foo/bar`
- `from foo import ...`
- `include "..."`

It does NOT yet pick up `{.importc, header: "...".}` pragmas — the
cross-language C-include dep is hand-authored. A follow-on milestone
will fold that into the scanner.

Run `repro deps refresh` to regenerate `repro.scanned-deps.nim`. Run
`repro deps refresh --check` in CI to gate the file's freshness.

## Outstanding limitations

- **External nimble deps.** Mode 3 has no `nimble` resolver — if you
  need a third-party package from `nimble.directory`, switch to Mode 2
  by adding a `<pkg>.nimble`.
- **`{.importc.}` header tracking.** The scanner doesn't yet read
  `header:` arguments out of `{.importc.}` pragmas. For
  cross-language deps, hand-author the `depends_on` edge until a
  scanner extension lands.
- **Multi-target Nim (compile to JS / ObjC).** The Mode 3 path emits
  C only. JS and ObjC backends remain through Mode 2 + `nim js` /
  `nim objc` invocations in a `build:` block.
- **Test discovery.** The Nim convention's test-target discovery is
  minimal today; for non-trivial test layouts, use Mode 2 (`nimble
  test`) or a `build:` block.
