# Getting Started

This is a five-minute walkthrough from cloning the reprobuild example
repo to running a built binary. You'll build a small two-package Nim
workspace consisting of a library (`greet`) and an executable (`hello`)
that imports it.

This page uses Nim because it's reprobuild's first-class Mode 3
language and Nim is already present in the dev shell. If you'd rather
use a different language, see the per-language tutorial under
[Languages](#next-steps).

## Prerequisites

You need:

- `repro` on `PATH` (the reprobuild CLI). See the [main
  README](../../README.md) for install instructions.
- A Nim toolchain on `PATH` (Nim 2.2 or newer). On reprobuild's dev
  shell this is automatically provisioned; on a stock host install via
  `choosenim` / `nimble` / your package manager.

Check the toolchain is wired up:

```text
repro --version
nim --version
```

Both should succeed before continuing.

## Step 1 — Clone the example

```bash
git clone https://github.com/metacraft-labs/reprobuild-examples
cd reprobuild-examples/nim/mode3-pilot
```

You should see this layout:

```text
nim/mode3-pilot/
  repro.nim
  repro.scanned-deps.nim
  mode3_pilot.nimble
  src/
    hello.nim
    greet.nim
```

## Step 2 — Look at the project file

Open `repro.nim`:

```nim
import repro_project_dsl

package mode3PilotGreet:
  uses:
    "nim >=2.2 <3.0"
  library greet

package mode3PilotHello:
  uses:
    "nim >=2.2 <3.0"
  executable hello:
    discard

# Workspace-internal deps regenerated by `repro deps refresh`.
include "repro.scanned-deps.nim"
```

That's the entire project description: two packages, each declaring
one member (a `library` or an `executable`), each saying "I use Nim
2.2+". No source paths. No compile flags. No link order. The Nim
convention figures all of that out from the source tree.

## Step 3 — Look at the sources

`src/greet.nim`:

```nim
proc greet*(name: string): string =
  "hello from mode3-pilot, " & name
```

`src/hello.nim`:

```nim
import greet

echo greet("mode3-pilot")
```

The `import greet` line in `hello.nim` is what the scanner picks up to
infer that `mode3PilotHello` depends on `mode3PilotGreet`. That
inference lives in `repro.scanned-deps.nim`:

```nim
# DO NOT EDIT — regenerated by `repro deps refresh`.
depends_on mode3PilotHello: mode3PilotGreet
```

## Step 4 — Build

From the `nim/mode3-pilot/` directory:

```text
repro build
```

That command:

1. Parses `repro.nim` (which `include`s `repro.scanned-deps.nim`).
2. Recognizes both packages as Nim Mode 3.
3. Emits the action graph: compile `greet` to a library, compile
   `hello` linking against `greet`'s archive.
4. Executes in topological order, caching each action's output.

Output binaries land under `.repro/build/`:

```text
.repro/build/
  greet/libgreet.a
  hello/hello[.exe]
```

## Step 5 — Run

```text
.repro/build/hello/hello
```

You should see:

```text
hello from mode3-pilot, mode3-pilot
```

That's the round trip: a workspace described in 14 lines of
`repro.nim`, with the scanner finding the `import` edge, the
convention emitting per-language compile + link actions, and the
engine sequencing them deterministically.

## Step 6 — Make a change

Edit `src/greet.nim`:

```nim
proc greet*(name: string): string =
  "hi there, " & name & "!"
```

Rebuild:

```text
repro build
```

The output changes accordingly. Run again to see the new greeting.
Notice that on the second build only `greet` and `hello` rebuild —
nothing else moves. That's incremental rebuilds working with the
hash-based action cache: only inputs whose content changed produce new
outputs.

## Step 7 — Add a dependency

Suppose `hello.nim` now needs a second library, `logger`. Add the
package:

```nim
# repro.nim
package mode3PilotLogger:
  uses:
    "nim >=2.2 <3.0"
  library logger
```

Add `src/logger.nim` exposing a `log` proc. Add `import logger` and
a call to `log(...)` to `hello.nim`. Refresh:

```text
repro deps refresh
```

The `repro.scanned-deps.nim` now contains:

```nim
depends_on mode3PilotHello:
  mode3PilotGreet
  mode3PilotLogger
```

(or two separate `depends_on` lines, equivalently). The scanner picked
up the new `import logger` line and wrote the edge for you.

Rebuild:

```text
repro build
```

The new edge is in effect.

## What just happened?

You built a Mode 3 project. The principles:

- **Tiny project file.** `repro.nim` only declares packages + uses.
- **Convention infers layout.** The Nim convention found
  `src/hello.nim` and `src/greet.nim` from the recognized layout
  without you naming them.
- **Scanner infers internal deps.** `repro deps refresh` walked the
  sources and wrote inter-package edges into a machine-authored file.
- **You stay in control.** Anything the scanner can't see (codegen
  steps, runtime plugin loads, etc.) you write by hand in `repro.nim`
  using the same `depends_on` macro the scanner emits.
