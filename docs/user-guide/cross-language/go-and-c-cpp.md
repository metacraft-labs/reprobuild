# Go ↔ C/C++ (cgo)

Go's cgo subsystem makes calling C from Go straightforward but
requires the full `go build` pipeline rather than the Mode 3
`go tool compile` / `go tool link` fast path. The reverse direction
uses Go's `c-archive` / `c-shared` build mode.

## Both directions in one paragraph

- **Forward (Go → C):** Go binary uses `import "C"` with a preamble
  comment containing `#include "mathlib/add.h"`. The convention
  switches from `go tool compile` to `go build` (cgo requires the
  full pipeline) and passes `-ldflags '-extldflags=-L<dir> -l<lib>'`
  to thread the upstream C archive onto the external linker.
- **Reverse (C → Go):** Go library compiled with
  `go build -buildmode=c-archive` produces a `lib<name>.a` archive
  plus a generated header. The C/C++ binary links against the
  archive and calls Go's runtime initializer before using exports.
  *(Status: partial — c-archive support is best-effort today.)*

## Minimal fixture (forward direction)

`repro.nim`:

```nim
import repro_project_dsl

package mathlib:
  uses:
    "gcc >=11"
  library mathlib

package calc:
  uses:
    "go"
  executable calc:
    discard

depends_on calc: mathlib

include "repro.scanned-deps.nim"
```

Layout:

```text
go-uses-cpp-lib/
  repro.nim
  repro.scanned-deps.nim
  mathlib/src/add.c
  mathlib/include/mathlib/add.h
  calc/go.mod                          # cgo needs go.mod for the module name
  calc/main.go
```

`calc/main.go`:

```go
package main

/*
#cgo CFLAGS: -I../mathlib/include
#include "mathlib/add.h"
*/
import "C"
import "fmt"

func main() {
    result := C.add(C.int(2), C.int(3))
    fmt.Printf("go says: mathlib added 2+3 = %d\n", result)
}
```

`mathlib/include/mathlib/add.h`:

```c
#ifndef MATHLIB_ADD_H
#define MATHLIB_ADD_H
int add(int a, int b);
#endif
```

`mathlib/src/add.c`:

```c
#include "mathlib/add.h"
int add(int a, int b) { return a + b; }
```

Build:

```text
repro build
```

`go-direct` takes ownership of the workspace. It emits the C archive
in-line (`gcc -c` + `ar rcs`), switches the Go binary's build from
`go tool compile` to `go build` since cgo requires the full Go
pipeline, and passes `-ldflags '-extldflags=-L<dir> -l<libname>'` so
the cgo external linker picks up the C archive. `CGO_ENABLED` defaults
to 1 in modern Go toolchains; `CC` resolves to `gcc` from `PATH`.

Reference fixture:
[`reprobuild-examples/mixed/go-uses-cpp-lib/`](https://github.com/metacraft-labs/reprobuild-examples/tree/main/mixed/go-uses-cpp-lib).

## Reverse direction (C → Go)

Reference fixture:
[`reprobuild-examples/mixed/cpp-uses-go-lib/`](https://github.com/metacraft-labs/reprobuild-examples/tree/main/mixed/cpp-uses-go-lib).

Go library:

```go
package main

import "C"

//export GoAdd
func GoAdd(a, b int32) int32 {
    return a + b
}

func main() {}    // required even though c-archive doesn't use it
```

Built with:

```text
go build -buildmode=c-archive -o libgolib.a
```

C/C++ binary:

```c
#include "libgolib.h"   // generated by go build

int main(void) {
    printf("GoAdd(2, 3) = %d\n", GoAdd(2, 3));
    return 0;
}
```

Note: c-archive output requires `pthread` + `dl` runtime libs on the
C link line.

## Required declarations

- **Go side, forward:**
  - `import "C"` (the cgo "package").
  - Preamble comment immediately above `import "C"` containing
    `#cgo CFLAGS: -I<path>` (the convention also threads the include
    path from the `depends_on` edge, but explicit is fine).
  - `#include "<header>"` to declare the C symbols.

- **Go side, reverse:**
  - `import "C"` + `//export <SymbolName>` directives above each
    exported function.
  - `package main` + an empty `func main()` (required for
    `c-archive` mode).

- **C side:** ordinary header / impl for forward direction; for
  reverse direction the consumer includes the generated header that
  cgo emits alongside the `.a`.

## Outstanding limitations

- **cgo is on the slow path.** When any package in the workspace uses
  cgo, the Go convention falls back from `go tool compile` to `go
  build`. Incremental rebuilds work but aren't as fine-grained.
- **Scanner blind to cgo preambles.** The Go scanner reads `import
  "module/path"` but doesn't fold `#include "..."` directives in cgo
  preambles into the dep graph.
- **Reverse direction (c-archive) is partial.** Works on Linux/macOS;
  Windows has known issues with `pthread` runtime linkage.
- **`go.mod` required for cgo.** Pure-Go Mode 3 (no `go.mod`) doesn't
  work with cgo. The cgo case effectively requires Mode 2 inside the
  Go package.
- **No `import "C"` codegen visibility.** The cgo-generated
  `_cgo_*.go` / `_cgo_*.c` files are produced inside `go build` and
  not exposed to reprobuild's action graph.
