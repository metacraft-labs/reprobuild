# Foreign-Package Runtime (C3)

This document describes the runtime architecture for foreign-distro
packages (apt/dnf/pacman) consumed via Reprobuild's content-addressed
store. It complements:

* [Linux-Third-Party-Distro-Packages.md](../Linux-Third-Party-Distro-Packages.md)
  (high-level design).
* [Linux-Third-Party-Sandbox-MVP.milestones.org](../Linux-Third-Party-Sandbox-MVP.milestones.org)
  (sandbox primitives, closed 2026-06-11).
* `ReproOS-Generations-And-Foreign-Packages.milestones.org` Phase C
  (harvesting + realize-time integration).

C3 ships:

* `apps/reprobuild-sandbox-launcher/launcher.c` -- the native
  user-namespace + bind-mount launcher (Linux); a no-op stub on
  Windows.
* `libs/repro_local_store/src/repro_local_store/sandbox_manifest.nim`
  -- realize-time manifest generator that walks the C2 catalog graph
  and emits a deterministic launcher manifest.
* `apps/reprobuild-sandbox-launcher/MANIFEST-FORMAT.md` -- the
  manifest file format.

## Architecture

```
                       harvest time                  realize time
                       (Phase C2)                    (Phase C3)
                                                   /---->  $prefix/launcher.manifest
                                                   |       (bind set + exec= path,
                                                   |        deterministic byte order)
                                                   |
recipes/catalog/foreign/apt/git.json  ->  walkCatalogGraph
       \-- dep_closure: [libc6, ...] -/  (libs/repro_local_store/
                                          sandbox_manifest.nim)         per-binary shim
                                                                       $prefix/bin/git
                                                                       (`exec` launcher)
recipes/catalog/foreign/apt/libc6.json
       \-- dep_closure: [libgcc-s1, ...] /

                                                            run time
                                                            (operator)
                                                            ----------
                                                            $ git --version
                                                              |
                                                              shim -> launcher
                                                                       |
                                                                       unshare(NEWUSER|NEWNS)
                                                                       mount(MS_BIND, ...) x N
                                                                       execve("$prefix/.../git", argv)
```

## Catalog graph walk

The C2 harvester emits one catalog file per package in the harvest
union. Each catalog carries ONLY its own per-record transitive
`dependency_closure`. The launcher's realize-time generator MUST walk
the catalog graph rooted at the package being installed -- it cannot
just read one root catalog's `dependency_closure` because:

* A future dnf/pacman harvester MAY emit non-transitive closures
  (immediate deps only); the consumer-side walker handles both.
* Composing two roots in one home profile (`git` + `vim`) means each
  binary needs its OWN root's closure -- separate per-root walks, not
  the harvest union.
* Per-record closure keeps the recursive `CacheEntryKey` composer's
  depth bounded.

See `walkCatalogGraph` in `sandbox_manifest.nim` for the
implementation. Failure modes:

* `CatalogResolveError` -- a dep file is missing under the catalog
  root. The walker hard-fails so the launcher never produces a hollow
  bind-set.
* `CycleDetectedError` -- the dep graph contains a cycle. apt cycles
  are rare but possible (mutually-recommending packages); we treat
  them as a hard error so the manifest is well-defined.

## Manifest format

The launcher reads a small line-oriented text format. See
[MANIFEST-FORMAT.md](../apps/reprobuild-sandbox-launcher/MANIFEST-FORMAT.md).

Determinism contract: identical inputs produce byte-identical
manifest text. The generator sorts the bind set by
`(target, source, flags)`; the writer normalizes path separators to
`/` so Windows-tested manifests parse cleanly under MSYS.

## Per-binary shim

For each binary in a foreign package's `bin/`, the realize pipeline
emits a tiny shim at `$prefix/bin/<binary>`:

```sh
#!/bin/sh
# Auto-generated by repro_local_store/sandbox_manifest.nim
exec /path/to/reprobuild-sandbox-launcher \
  --manifest=$prefix/launcher.manifest \
  --exec=/actual/store/path/to/<binary> \
  -- "$@"
```

Putting `$prefix/bin` on `PATH` (via the home-profile activation
pass) gives the operator a transparent launcher. The wrapped binary
sees an FHS-coherent filesystem view inside the namespace.

## Launcher behavior (Linux)

1. Parse the manifest (raw text I/O, no allocator runtime overhead).
2. `unshare(CLONE_NEWUSER | CLONE_NEWNS)`. The user namespace lets us
   `mount(2)` without root; the mount namespace scopes the binds to
   the current process tree.
3. Write `/proc/self/uid_map` + `/proc/self/gid_map` so the
   unprivileged caller appears as root *inside* the namespace --
   `mount(2)` requires CAP_SYS_ADMIN.
4. `mount(MS_REC | MS_PRIVATE)` on `/` so subsequent binds don't
   leak back to the host.
5. For each manifest entry, `mount(MS_BIND, ...)`; for `ro` entries a
   second `mount(MS_REMOUNT | MS_RDONLY)` (Linux ignores `MS_RDONLY`
   on the initial bind).
6. `chdir(cwd)` if requested.
7. `execve(exec_path, argv)`.

## Launcher behavior (Windows)

The launcher parses the manifest and `_execv()`s the target binary
directly. No namespace setup happens because Windows lacks Linux's
user-namespace + bind-mount machinery. Reprobuild's foreign-package
sandbox is Linux-only; the Windows stub exists so cross-platform
tooling (manifest emit + CI smoke) can exercise the parser path.

## Threat model

* The launcher does NOT modify the host filesystem. All bind mounts
  live inside the new mount namespace and disappear when the
  namespace dies (process exit).
* The launcher does NOT preserve setuid bits or file capabilities --
  unprivileged user namespaces drop these per kernel policy. A
  foreign package that needs setuid (e.g. `ping` with `cap_net_raw`)
  will run without those capabilities. This is acceptable for the
  Tier-3 use cases (operator's own tools); production system services
  belong on Tier 1 / Tier 2.
* Signature trust is still Tier 3: the binary is byte-identical to
  the distro's upstream bytes (validated at harvest time via
  `InRelease` GPG), but Reprobuild does not re-verify per-invocation.
  The catalog file itself is consumed on trust at realize time --
  per-catalog signature verification is deferred (see C2 risk #3 in
  the campaign spec).
* The launcher's `--dry-run` lets operators audit the manifest before
  the unshare; combined with `--verbose` it surfaces every bind
  operation.

## Performance

The C3 spec stipulates sub-100ms launcher overhead. Measured on WSL2
Ubuntu 22.04 (kernel 6.18, x86_64):

| metric              | observed |
| ------------------- | -------- |
| p50 wall-clock      | 3 ms     |
| p95 wall-clock      | 4 ms     |

(See `tests/integration/foreign_packages/t_c3_launcher_perf.sh`.)
The overhead is well under budget because the launcher is a single
process with no allocator runtime, no extra `fork()`, and a small
number of `mount(2)` calls.

## Limitations

* **Windows**: the launcher is a no-op stub. Foreign packages can't
  be realized on Windows; this is by design.
* **macOS**: unsupported. macOS lacks Linux's bind-mount machinery;
  the Tier-3 sandbox is Linux-only.
* **Static binaries inside the namespace**: bind-mounting a fresh
  `/lib` over the host's hides the dynamic linker. Tooling that lives
  inside the wrapped binary's environment (e.g. the wrapped binary's
  own `#!/bin/sh` interpreter) must come from the foreign-package
  closure, not the host. Statically linked probes work without any
  bind set.
* **setuid / file capabilities**: stripped (see threat model above).
* **`/etc/resolv.conf` / `/etc/passwd` / `/etc/group`**: not
  bind-mounted by default. Tools that need them must either declare
  the bind in their package's catalog or operate without (DNS via
  `dig +noresolv`, etc.). This is the same trade-off NixOS's
  `buildFHSEnv` makes.
* **GUI applications**: X11 / Wayland sockets, GPU device nodes,
  audio servers are not exposed. Scoped out of the MVP.
