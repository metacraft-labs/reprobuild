## Phase A — characterize the 404

M9.R.54 landed devShells.default runtime seeding (qemu, grub2_efi,
kmod) but the M9.R.54.3 close-out noted `nix develop` itself was
unreachable because the ``codetracer-native-recorder`` flake input
pinned to rev ``505b7433faf278bbf8a1b7dc297418f501baea7d`` returned
HTTP 404 from the tarball archive URL:

    $ curl -sI https://github.com/metacraft-labs/codetracer-native-recorder/archive/505b7433faf278bbf8a1b7dc297418f501baea7d.tar.gz
    HTTP/1.1 404 Not Found

    $ curl -sI https://codeload.github.com/metacraft-labs/codetracer-native-recorder/tar.gz/505b7433faf278bbf8a1b7dc297418f501baea7d
    HTTP/1.1 404 Not Found

Falsifying the "repo is private / deleted" hypothesis (which M9.R.54
had recorded as the presumed cause), the git wire-protocol endpoints
all resolve fine, both against the exact pinned rev *and* against
the ``stable`` branch tip, anonymously (no PAT, GIT_TERMINAL_PROMPT=0):

    $ GIT_TERMINAL_PROMPT=0 git ls-remote https://github.com/metacraft-labs/codetracer-native-recorder.git refs/heads/stable
    505b7433faf278bbf8a1b7dc297418f501baea7d	refs/heads/stable

    $ GIT_TERMINAL_PROMPT=0 git clone --depth=1 --branch=stable \
        https://github.com/metacraft-labs/codetracer-native-recorder.git ctnr-test
    Cloning into 'ctnr-test'...
    # succeeds; hook_registry.nim etc. present under ct_interpose/src/ct_interpose/

The GitHub REST API also 404s for the individual commit
(``api.github.com/repos/.../commits/<rev>``).  Cross-checked against
the HEAD-of-main rev (``b9725264...``) — same 404 shape — so the
tarball / API endpoints are broken across every rev of this repo, not
just the pinned one.  Root cause: the tarball archive endpoint
(``codeload.github.com``) and the REST commit endpoint are apparently
disabled for this repository (likely a repo-level setting), while the
git wire protocol remains fully functional.  Nix's ``github:`` URL
form uses codeload; ``git+https:`` uses the git wire protocol.


## Phase B — option chosen + rationale

Four options considered per the plan:

  1. Latest working upstream rev via ``git ls-remote`` — falsified
     the "gone upstream" hypothesis (repo exists, refs resolve), so
     the pin itself isn't stale.  Not the fix.
  2. Sibling repo (``path:../codetracer-native-recorder``) — works
     for the dev shell but breaks the sandboxed package build (no
     sibling inside the sandbox), and locks the flake to a
     host-relative path that fresh checkouts can't resolve.  Would
     require ``--override-input`` on CI.  Not clean.
  3. Vendor last-known-good tree — heavy for a repo whose git
     endpoints are still fully functional.
  4. Drop the input — falsified: reproos-image doesn't directly
     depend on ct_interpose, but the flake's package build wires
     CT_INTERPOSE_SRC unconditionally (``ctInterposeSrc = "${...}"``
     at flake.nix:134), and repro_monitor_hooks + the cross-platform
     monitor shim import from it, so dropping breaks the package
     build.  Not viable without a much larger refactor.

Chosen: **swap URL form from ``github:`` to ``git+https:``**,
staying pinned to the same rev.  Cleanest — one-liner ``url =``
change, no ``--override-input`` / ``--impure`` escape hatch, works
from a fresh checkout, and Nix's ``git+https`` fetcher produces a
byte-identical narHash to the previously-locked tarball
(``sha256-wNr+jtXbOWYeWhiOpHR8jYP8+KnxZbeiaOYmB/EUljI=`` unchanged
in flake.lock across the swap).
