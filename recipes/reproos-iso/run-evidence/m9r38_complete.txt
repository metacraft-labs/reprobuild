EXECUTIVE SUMMARY
=================

The M9.R.37 chain landed three carry items: M9.R.37.6 boot-installed
driver, M9.R.37.7 GRUB serial-console wiring, M9.R.37.8 vmlinuz path
fix.  M9.R.38 was supposed to rebuild the ISO with those three fixes,
re-run install/boot to capture the installed-system DE smoke
transcript.

G1 succeeded after a recipe-wiring fix: ``recipes/reproos-iso/``'s
``extraInputs`` lists the installer BINARY (not source), so a fresh
``installer_state.cpp`` doesn't trigger an installer rebuild via the
ISO recipe.  M9.R.38.1 added an explicit
``_m9r38_rebuild_installer.sh`` driver that invokes
``repro build apps/reproos-installer`` directly, plus a verifier that
greps the installer binary for the M9.R.37.7+8 GRUB-emit strings.
Two clean rebuilds (00:11 + 00:51 + 00:57) produced bit-identical
binaries with both fixes baked in.

G2 hit a HARD WALL: the freshly-built installer aborts with
``munmap_chunk(): invalid pointer`` very early in Qt init (right
after the locale C->C.UTF-8 message, before Phase 1).  Same crash
under both the clean driver (_m9r38_install.sh) and the diagnostic
driver (_m9r37_install_diag.sh) -- the strace tail shows the
launcher exec'd ``stdbuf -oL -eL /usr/bin/reproos-installer
--automated /etc/reproos/auto-config.toml``, then strace's child
waited on do_wait and the installer aborted before Phase 1.

The crash is SPECIFIC to the live ISO env:

  * On the host (eli-wsl, outside QEMU), the EXACT SAME binary at
    ``apps/reproos-installer/.repro/output/install/usr/bin/reproos-installer``
    runs cleanly under ``QT_QPA_PLATFORM=offscreen ... --automated``
    -- gets to Phase 1, fails on ``spawn failed: ... execve: No such
    file or directory`` because ``repro`` isn't on PATH in the host
    nix-shell.  No munmap.

  * On the host via the launcher (with a symlink at
    ``/usr/bin/reproos-installer``), same thing -- Phase 1 reaches
    repro spawn, no munmap.

  * Inside the live ISO QEMU, immediate munmap_chunk + SIGABRT
    (exit 134 = 128 + 6) before any Phase output.

ABI MISMATCH HYPOTHESIS (TESTED, FALSIFIED AS PRIMARY CAUSE)
-------------------------------------------------------------

The de-rootfs mirror contains TWO Qt6 versions:

  * ``/nix/store/zp6r9bxds...-qtbase-6.10.1/`` -- Qt 6.10.1
  * ``/opt/repro/.../qt6-base/.repro/output/install/usr/lib/`` -- 6.8.1

Inspection showed the M9.R.36.1/M9.R.37.5 launcher walked
``/nix/store/*/lib`` for any ``qt-6/plugins`` dir and added it to
QT_PLUGIN_PATH / QT_QPA_PLATFORM_PLUGIN_PATH -- which would have
matched qtbase-6.10.1 and caused Qt 6.8.1 binary to load 6.10.1
plugins (C++ ABI mismatch -> heap corruption -> munmap_chunk).

M9.R.38.3 fixed the launcher to:

  1. SKIP all ``/nix/store/*-qt*-*/lib`` dirs in the qt-6/plugins +
     qt-6/qml walk.
  2. EXPLICITLY add ``/opt/repro/.../qt6-base|qt6-declarative|
     qt6-quickcontrols2|qt6-tools/.repro/output/install/usr/lib/
     qt-6/{plugins,qml}`` after the nix-store walk.

ISO rebuild #4 (01:31) staged the M9.R.38.3 launcher.  G2 install
attempt #4 (01:35) STILL CRASHED with the same munmap signature.
That falsifies the launcher-qt-mix hypothesis as the PRIMARY cause
-- since the launcher's plugin discovery now only includes 6.8.1
paths and the crash happens BEFORE any Qt platform plugin loads
(the locale C->C.UTF-8 message is QCoreApplication's init, plugin
load happens later under QGuiApplication).

Possible remaining causes (not yet diagnosed):

  * libstdc++.so.6 mismatch -- the live ISO's Debian Trixie ships
    libstdc++.so.6 from gcc-12; the installer was linked against
    gcc-14.3.0-lib's libstdc++ via RPATH last entry.  If ld.so
    resolves libstdc++ via the Debian path (system loader cache)
    before the RPATH entry, the binary calls C++ runtime built for
    14 against runtime version 12 -> ABI gap on any new vtable.
    LD_DEBUG=libs would confirm.

  * Qt6Core 6.8.1's QObject vtable layout vs the libraries it
    transitively pulls in -- if libQt6Network or libQt6Concurrent
    resolve to 6.10.1 nix-store paths (the launcher's skip pattern
    covers qtbase-* but possibly not all transitive subpaths the
    rpath isn't fully patched on).

Confirming requires running ``LD_DEBUG=libs strace -e openat
/usr/bin/reproos-installer --automated ...`` inside the live ISO
to see EVERY shared-lib resolution decision the loader makes.  The
M9.R.37.1 diag harness writes /tmp/installer.strace inside the
ephemeral tmpfs that vanishes on poweroff; the next iteration needs
to ``cp`` the trace to /mnt (post-Phase-3 mount) BEFORE poweroff so
the trace survives.
