### G2 (M9.R.32.2) — gnome-shell mozjs blocker: DEFERRED (documented)

gnome-shell's install mirror has never been populated.  The build
requires ``gjs`` (the GNOME JavaScript engine) at meson configure time
via ``js_min_version`` + the ``gjs-1.0`` pkg-config probe.  ``gjs`` is
declared as a buildDep but exists only as a Nix-channel stub
(``libs/repro_dsl_stdlib/src/repro_dsl_stdlib/packages/gjs.nim``) —
not a from-source recipe.

gjs itself depends on SpiderMonkey (``mozjs-128`` or newer); building
SpiderMonkey from-source is a multi-day effort (Rust toolchain + clang
≥15 + Python 3.10 + cbindgen + 1.5 GiB Mozilla-central source tree).
Authoring both gjs + mozjs from-source is out of M9.R.32 scope.

The recipe header now carries an honest STATUS docblock so a future
agent doesn't burn time re-discovering the blocker.  v1 ships GNOME
via mutter alone (which DOES build from-source);
the GNOME .desktop entry in stage-de-rootfs.sh remains but gnome-shell's
startup will fail at runtime until a future fullbuild milestone
(M9.R.34?) authors gjs + mozjs.

### G3 (M9.R.32.3) — reproos-installer-launcher missing: CLOSED (verified)

The M9.R.31 evidence reported ``execve: No such file or directory`` for
``/usr/bin/reproos-installer-launcher`` at SDDM auto-launch.  Direct
squashfs inspection of the M9.R.31 ISO showed otherwise:

  $ nix-shell -p squashfsTools --run "
      sudo unsquashfs -l recipes/reproos-iso/build/reproos.iso |
      grep reproos-installer
    "
  squashfs-root/etc/profile.d/zz-reproos-installer-autostart.sh
  squashfs-root/usr/bin/reproos-installer
  squashfs-root/usr/bin/reproos-installer-launcher
  squashfs-root/usr/share/wayland-sessions/reproos-installer.desktop

Extracted launcher (808 bytes) is a sway-in-kiosk shell script;
extracted ``/etc/sddm.conf.d/00-autologin.conf`` points at the right
session:

  [Autologin]
  User=live
  Session=reproos-installer
  Relogin=true

Extracted ``/usr/share/wayland-sessions/reproos-installer.desktop``
points at the right launcher:

  [Desktop Entry]
  Type=Application
  Name=ReproOS Installer
  Comment=First-boot ReproOS installer wizard (kiosk mode)
  Exec=/usr/bin/reproos-installer-launcher
  DesktopNames=reproos-installer

The M9.R.31 ``execve: No such file or directory`` report was thus
either a transient sequencing artefact (SDDM probed the path before the
squashfs mounted) or a stale-ISO mis-report from a prior build that
preceded M9.R.18.14's launcher addition.  No code change is needed.

stage-de-rootfs.sh's launcher script (lines 358-396) + the SDDM
autologin block (lines 484-495) remain the source of truth.  G5's
Phase D smoke (when run) will re-validate end-to-end.

### G4 (M9.R.32.4) — apt audit: CLOSED (no entries removable yet)

Annotated every line of ``recipes/reproos-iso/scripts/build-base-rootfs.sh``
PKG_LIST with FS:done|partial|none + STAGE:yes|no markers + a
DROP-CRITERION docblock.

Audit result: NO apt entry can be safely removed at M9.R.32.
Every from-source replacement that EXISTS (FS:done — systemd,
util-linux, kmod, dbus, sudo, e2fsprogs, btrfs-progs, shadow-utils,
iana-tzdata) is NOT mirrored into the de-rootfs by stage-de-rootfs.sh
(STAGE:no — the script only mirrors the DE entry-point binaries:
sway, kwin, mutter, sddm, plasma-workspace, gdm).

The audit makes the M9.R.33 work shape explicit:

  * Extend stage-de-rootfs.sh with a per-recipe base-mirror loop that
    walks every PKG_LIST entry's from-source install/usr/{bin,sbin,
    lib,lib64,share} tree and mirrors it into the rootfs.
  * Once that lands, the apt PKG_LIST shrinks to ONLY the FS:none
    entries (data packages + recipes not yet authored).

Per-entry FS state was measured 2026-06-24 by inspecting each
``recipes/packages/source/<recipe>/.repro/output/install/`` tree.

NB: this commit invalidates the base-rootfs cache (script-body hash
participates in CACHE_KEY by design, M9.R.29.17).  The next ISO build
will rebuild the trixie-slim tarball from upstream debian:trixie-slim.

### G5 (M9.R.32.5) — ISO rebuild + Phase E DE smoke: BLOCKED

Discovered during the M9.R.32.1 verification run: a fresh plasma-
workspace ``cmake configure`` (after ``rm -rf build/``) trips on
multiple latent issues that the prior M9.R.31 build dodged by inheriting
a stale-but-working CMakeCache.txt:

  1. GLESv2 + PlatformGraphics — addressed by M9.R.32.1.2 by threading
     the qt6-base-shipped Qt6 platforms cmake-module dir into
     CMAKE_MODULE_PATH + baking the mesa GLESv2 hints.
  2. QCoro6 — fresh configure fails with ``QCoro6 not found``.  qcoro6
     is a Nix-channel stub (``packages/qcoro6.nim`` points at
     ``nixpkgs#kdePackages.qcoro``), not a from-source recipe.  The
     prior CMakeCache.txt had a Nix-store-baked ``QCoro6_DIR`` that
     somehow resolved at the M9.R.30 build; that path is gone with the
     wipe.  Surfacing QCoro6 cleanly under ``--tool-provisioning=
     from-source`` is an unresolved architectural gap.

Architectural conclusion: the plasma-workspace recipe inherited a
"works only with cached CMakeCache.txt" state from M9.R.30's bulk
rebuild.  A truly-from-scratch rebuild requires multiple additional
fixes (from-source qcoro6 recipe OR auto-thread Nix-stubbed cmake-
config dirs, plus the M9.R.15i.5 walker extension for
CMAKE_MODULE_PATH).  All M9.R.33 work shape.

The deferred-to-M9.R.33 work:

  1. Either author a qcoro6 from-source recipe (~half a day; QCoro is
     a small Qt6 coroutines wrapper, no heavy deps beyond Qt6 itself),
     OR extend the M9.R.15i.5 walker to fetch Nix-stub-provisioned
     cmake-config dirs and thread them as
     ``-D<Component>_DIR=<nix-store-path>``.
  2. Extend cmake_package's M9.R.15o-family walker to thread
     CMAKE_MODULE_PATH from every Qt6* dep's ``cmake/Qt6/platforms/``
     dir AND surface GLESv2 from mesa's pkg-config.  M9.R.32.1.2's
     recipe-local fallback can then be dropped.
  3. With both landed, ``rm -rf build && repro build`` should succeed
     end-to-end on plasma-workspace.
  4. Then run ``_m9r32_iso_build.sh`` →
     ``_m9r32_qemu_de_smoke.sh`` → installed-system Phase E.

The live ISO from M9.R.31 (built 2026-06-24 12:20) WORKS for sway /
kwin / mutter / plasmashell / sddm-greeter-qt6 as documented in
``m9r31_complete.txt``; the M9.R.32 driver scripts in repo will

BUILD WEDGE INCIDENT
====================

During the M9.R.32.1 verification run, the plasma-workspace cmake
build wedged for approximately 30 min waiting on a dependency (cmake's
make/-j8 build dispatched ``cmake_autogen`` for klipper which exited
fine, but the parent waited indefinitely on what appeared to be
stale kcoreaddons defunct moc child processes from a session started
at 09:28 that day).  After diagnosing the issue I cleared the
stale processes; in doing so I inadvertently killed the
plasma-workspace build entire.  Restart kicked at 13:28.

The wedge pattern is consistent with the documented M9.R.15q.13.6
cmake-AUTOGEN multi-pipe race, BUT here the wedge was inherited from
a sibling recipe's leftover defunct processes, not from
plasma-workspace's own AUTOGEN.  Recommended follow-up: a sanity
``pkill -KILL`` for orphan defunct ``moc`` processes at the start of
each ``repro build`` invocation.
