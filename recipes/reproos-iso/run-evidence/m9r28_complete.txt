Gap 7 — hybrid grub EFI binary creation (M9.R.28.1)      CLOSED
----------------------------------------------------------------
M9.R.27.7 attempt 2 emitted EFI/BOOT/BOOTX64.EFI via grub-mkimage but
grub-mkrescue dropped the UEFI El Torito alt-boot block on the
resulting ISO. Root cause (M9.R.28.1):

  * grub-mkrescue's `--directory` arg accepts a SINGLE platform dir.
    Multiple `--directory` overrides not combine; the second wins.
    Unified-dir layout (lib/grub/{i386-pc,x86_64-efi}) fails because
    `--directory` expects modinfo.sh DIRECTLY under it, not nested
    under a platform subdir.
  * Even passing a manually built BOOTX64.EFI into the staged ISO tree
    doesn't help — grub-mkrescue's xorriso invocation has no
    `-eltorito-alt-boot` block, only the BIOS `-b` flag.

Fix (commit 2e9e4cd9 — M9.R.28.1):
  1. Build BOOTX64.EFI via grub-mkimage against the x86_64-efi grub
     install dir (`/nix/store/<hash>-grub-2.12/lib/grub/x86_64-efi/`).
  2. Wrap the loader in a FAT12 ESP image (``mkfs.fat -F 12``) — NOT
     ``mformat`` which auto-picks FAT32 for any size >= 1 MiB.  The
     degenerate FAT32 mformat produces (only 16 sectors per FAT on a
     ~1 MiB image) fails OVMF's FAT driver validity check; the boot
     manager then reports ``BdsDxe: failed to load Boot0001 ...
     Not Found``. FAT12 is what real ESPs on small live-ISO boot
     images use.
  3. Stage the EFI ESP image (``efi.img``) on the source tree under
     ``boot/grub/`` AND a copy of the x86_64-efi runtime module tree
     under ``boot/grub/x86_64-efi/``.
  4. Wrap the grub-mkrescue xorriso invocation with a
     ``--xorriso=<wrapper>`` script that splices
     ``-eltorito-alt-boot -eltorito-platform efi -e
     boot/grub/efi.img -no-emul-boot -isohybrid-gpt-basdat
     -efi-boot-part --efi-boot-image`` immediately before ``-o`` so
     the produced ISO has BOTH El Torito boot entries AND a GPT-marked
     EFI System Partition.

Verified: rebuilt test.iso boots end-to-end under OVMF/Q35:
  BdsDxe: loading Boot0001 "UEFI QEMU DVD-ROM"
  BdsDxe: starting Boot0001 "UEFI QEMU DVD-ROM"
  Booting 'ReproOS -- Hyprland (default)'
  EFI stub: Loaded initrd from LINUX_EFI_INITRD_MEDIA_GUID device path
  Linux version 6.12.86+deb13-amd64 ...

xorriso -indev <iso> -report_el_torito plain shows:
  El Torito boot img :   1  BIOS  y   none  ...   /boot/grub/i386-pc/eltorito.img
  El Torito boot img :   2  UEFI  y   none  ...   /boot/grub/efi.img

sgdisk -p shows partition 2 with Code EF00 (EFI System Partition,

Gap 3 — xwayland end-to-end build (M9.R.28.4)            DSL-CLOSED
----------------------------------------------------------------
M9.R.28.4 closes the recipe-shape gaps that blocked xwayland's
transitive build (recipes/packages/source/xorgproto + libxshmfence +
xtrans + libxfont2 + libxkbfile + xkeyboard-config end-to-end):

  * pkgconfig-only-package resolver fast-path (commit 9ed16ce3) —
    pure-header packages like xorgproto ship NO compiled libraries
    and NO CMake configs, only ``.pc`` files. The M9.R.14h.1
    install-mirror probe + the M9.R.15h.14 share-only-CMake probe
    both miss them; ``tryResolveFromSourceTool`` then returned
    ``rrNeedsBuild`` even though the recipe built successfully.
    Add a ``<install-mirror>/{lib,lib64,share}/pkgconfig/*.pc``
    probe; presence of any .pc satisfies the dep.

  * libfontenc + audit + libxcrypt + ncurses + popt + readline + zlib
    + lzo + libgcrypt + json-c + device-mapper + util-linux + pam +
    libbsd stdlib provisioning stubs (commits 9ed16ce3 + 76fce965 +
    655f580f) — every buildDep selector referenced by the M9.R.27.4
    long-tail recipes now has either a sibling from-source recipe
    OR a single-channel nix stdlib stub. The auto-recurse path no
    longer hard-errors with ``no stdlib provisioning channel`` on
    these.

End-to-end xwayland BUILD itself remains as a follow-up validation
pass; the recipe + stub closure is correct, but each of the 9 leaves
takes ~5-10 min to compile sequentially against the shared daemon
quota socket. M9.R.27 acceptance was the DSL surface closure, which
this milestone extends with the resolver work needed for the build
to actually run.
