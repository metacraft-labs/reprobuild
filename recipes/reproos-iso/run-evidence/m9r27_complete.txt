Gap 5 — nix-develop cachix 401 (M9.R.27.5)               CLOSED
----------------------------------------------------------------
Investigation results:

- The 401 originates from the per-user `~/.config/nix/nix.conf`
  on eli-wsl which lists `metacraft-labs-codetracer.cachix.org`
  as a substituter.
- The flake.nix doesn't reference cachix; the system-wide
  /etc/nix/nix.conf only lists cache.nixos.org.
- The 401 is encountered whenever `nix eval`, `nix develop`, or
  `nix-build` invokes a substituter that requires authentication.
- The iso build chain uses `nix-shell -p <pkg>` (not `nix develop`),
  which DOES inherit per-user substituters. The 401 appears
  occasionally in stderr but doesn't block the build (substituters
  are tried in declared order; cache.nixos.org succeeds for every
  needed derivation).
- Verified by running `repro build recipes/reproos-iso` under
  `nix-shell -p ...` — proceeds past the 401-emitting probe to
  satisfy from cache.nixos.org.

Fix: no code change required. The 401 is benign provided
cache.nixos.org is reachable.

Gap 7 — OVMF + QEMU bisect for UEFI boot (M9.R.27.7)     CLOSED
----------------------------------------------------------------
ROOT CAUSE IDENTIFIED + FIX LANDED.

Bisect harness (`m9r27_uefi_bisect.sh`, commit 8ee3f307 + the
chmod fix de0129b9) runs 6 OVMF + machine + display + serial-
routing combinations, captures stdout+stderr + serial transcript
per combo. Results on the M9.R.25-era ISO:

  combo1 (i440fx + display none):                0 bytes
  combo2 (q35 + display none):                 317 bytes
  combo3 (q35 + vga std + display none):       317 bytes
  combo4 (q35 + vga std + nographic):          317 bytes
  combo5 (i440fx + vga std + nographic):         0 bytes
  combo6 (q35 + virtio-vga + sdl):             317 bytes

Transcript analysis (combo2 etc.):
  BdsDxe: failed to load Boot0001 "UEFI QEMU DVD-ROM QM00005 "
    from PciRoot(0x0)/Pci(0x1F,0x2)/Sata(0x2,0xFFFF,0x0): Not Found
  BdsDxe: No bootable option or device was found.
  BdsDxe: Press any key to enter the Boot Manager Menu.

ROOT CAUSE: the ISO is BIOS-only. `xorriso -indev <iso>
-report_el_torito as_mkisofs` shows only
``-b /boot/grub/i386-pc/eltorito.img`` (the BIOS El Torito boot
catalog entry) — NO `-eltorito-alt-boot -e ... -no-emul-boot`
block for a UEFI ESP image. The script (`build-iso.sh`)
invokes `grub-mkrescue` expecting it to auto-detect both
arches; on Nix-based hosts the `nixpkgs#grub2` derivation
ships ONLY `i386-pc` modules and `nixpkgs#grub2_efi` ships
ONLY `x86_64-efi` modules (two separate /nix/store outputs).
grub-mkrescue's default `--directory` is
`$(dirname $0)/../lib/grub` which sees only one arch.

FIX (commit c9e6a462): in `build-iso.sh`, assemble a private
merged grub module dir ($WORK/grub-modules) that symlinks
BOTH `i386-pc` and `x86_64-efi` trees (the missing arch
probed from /nix/store/*-grub-2.*/), then pass
`--directory=$WORK/grub-modules` to grub-mkrescue. Logs
`GRUB_HAS_BIOS=1 GRUB_HAS_EFI=1` when both archs are present
and warns if only one is.
