PHASE B: ROOT CAUSE
====================

The M9.R.39.6 launcher set ``-E LD_LIBRARY_PATH=$_repro_ldpath``
on strace's env-export list, which puts ``LD_LIBRARY_PATH`` in the
TRACED CHILD's env (the installer).  M9.R.39 needed this to fix
the heap-corruption SIGABRT (both halves of glibc had to resolve
through the same nix instance).

The installer is a Nim binary; ``execCmdEx`` calls ``startProcess``
which forks + execs ``/bin/sh -c <cmd>``.  ``/bin/sh`` is Debian's
``dash`` binary on the live ISO, PT_INTERP'd at
``/lib64/ld-linux-x86-64.so.2`` (Debian's ld).  ``/bin/sh`` inherits
the parent's env including ``LD_LIBRARY_PATH``.  Debian ld walks
``LD_LIBRARY_PATH`` BEFORE ``/etc/ld.so.cache`` when resolving
``libc.so.6``.  The first hit in the launcher's prepended nix-glibc
dir is ``glibc-2.40-66/libc.so.6``.

dash was linked against Debian glibc-2.41's ``libpthread.so.0``,
which exports ``__nptl_change_stack_perm`` at GLIBC_PRIVATE.  Nix
glibc-2.40-66's ``libpthread.so.0`` doesn't export that symbol --
the GLIBC_PRIVATE symbol set drifts every minor version.  Loader
fails the relocation -> dash exits 127 with the symbol-lookup-error
text on stderr.

``execCmdEx`` merges stderr into stdout via ``poStdErrToStdOut``;
the caller (probeFilesystems) sees the symbol-lookup-error text
as the "lsblk JSON output" and ``parseJson`` fails at byte 0.

The bug is not in lsblk itself -- lsblk never ran.  The bug is
that ``LD_LIBRARY_PATH`` set for the installer's nix-glibc
resolution leaks into the env of every Debian subprocess the
installer spawns.

PHASE C: FIX
=============

Two layers of defence (M9.R.40.1):

1. Launcher level (a): mirror the non-DIAG branch's existing trick.
   Invoke ``ld.so --library-path`` AS the strace child.  ld.so
   processes ``--library-path`` itself; the kernel exec path
   ignores ``LD_LIBRARY_PATH`` (it's not in env); the installer's
   children get a clean env without ``LD_LIBRARY_PATH`` and Debian
   binaries (/bin/sh, lsblk, mount, findmnt, lspci) keep using
   Debian glibc.

2. Probe level (b): even with the launcher fix, the
   ``runOrEmpty`` wrapper now uses ``execShellCmdCleanEnv`` which
   strips every LD_* env var.  Forward-compatible against any
   future launcher change that re-introduces propagation.

The util-linux bare-``bin/`` mirror (c+d) is a separate concern but
related: lsblk on the live ISO had been silently resolving to
apt-installed Debian lsblk because the from-source one was missing
from the ``install/`` tree.  The mirror fix routes ``/usr/bin/lsblk``
to a from-source shadow-link in the staged ISO, matching the M9.R.33
"from-source-first" architectural principle.

====================

downstream gap -- is CLOSED.  The installer completes all 6 phases

The next layer down is the M9.R.24 install-time ``repro infra
apply --target`` stub.  The installer flags this explicitly:

  Phase 5: applying system profile...
  $ /usr/bin/repro infra apply --target /mnt
    repro infra apply: unknown flag: --target
  system apply (`repro infra apply --target`) is stubbed for the
  M9.R.24 demo; proceeding with a minimal bootable-system bootstrap

The minimal bootstrap copies the live ISO's vmlinuz + initrd into
``/mnt/boot/`` and installs GRUB -- but there's no Debian root on
the installed disk (no /bin /usr /etc/* beyond a tiny stub).  The
installed system can't boot to a usable shell, let alone a DE.

Closing the install -> boot -> DE-smoke loop end-to-end requires
M9.R.41+ to unstub ``repro infra apply --target``, which is a
materially different task than the M9.R.39/40 character + fix
cycle: it needs the apt-rootfs squashfs (or a from-source root)
to be copied onto /mnt + the initramfs regenerated for the
installed root rather than the live medium.

The M9.R.40 milestone closes its own gap (the hardware probe
JSON parse error) + hands off the NEXT gap (the M9.R.24 apply
stub) with full evidence + the M9.R.40.3 boot-installed driver
ready for the next investigator to wire up against a fully-staged
installed disk.
