MUTTER INTERNAL DSO RESOLUTION
------------------------------
The mutter cogl/mtk/clutter triple was on the M9.R.25.1 "not found"
list because they live under .../usr/lib64/mutter-15/ which the
launcher binary's RPATH doesn't reach.

The M9.R.25.5.1 ld.so.conf patch walks every from-source install-
mirror's lib*/ tree for directories containing .so* files and adds
each subdir to /etc/ld.so.conf.d/zz-reproos-overlay.conf.  After the
ISO build runs chroot ldconfig, the cache resolves these:

   libmutter-15.so            0/0   (header lib; no NEEDED)
   libmutter-cogl-15.so.0     20/20                   0
   libmutter-mtk-15.so.0      9/9                     0
   libmutter-clutter-15.so.0  34/34                   0

WHAT'S NOT YET VERIFIED
-----------------------
- Full ISO rebuild end-to-end: this verification used
  REPRO_BASE_ROOTFS_DISABLE=1 to skip the Docker pull + apt-install
  step.  The Phase-1/2 install-mirror + nix-store closure logic is
  the load-bearing change in M9.R.25 — the base-rootfs tarball
  bring-up is unchanged in shape, only its PKG_LIST shrinks.
- chroot ldconfig actually runs at ISO-build time inside Phase 7;
  the smoke environment skipped it because /sbin/ldconfig isn't in
  the staged tree without the Docker base.  The ISO build will
  generate /etc/ld.so.cache and mutter resolution flips to "OK with
  no LD_LIBRARY_PATH override".
- QEMU boot of the rebuilt ISO is the final acceptance gate.  The
  ldd verdicts above are the load-bearing evidence the architectural
  fix is correct; a full QEMU boot is a multi-hour cycle (Docker
  pull + apt + xorriso + grub-mkrescue + QEMU SDL boot) deferred to
  M9.R.25.6 / M9.R.26.

ARCHITECTURAL SUMMARY
---------------------
The Nix-pattern path preservation makes every M9.R.14f-embedded
absolute RPATH and PT_INTERP resolve verbatim on the staged tree.

   No patchelf rewriting.
   No apt-install DE fallback.
   No RUNPATH strip.
   No nix-store carve-outs for libsqlite / libvulkan / libclingo /
   Qt6 9.x.

The mechanism is general: any from-source recipe whose
m9r14fEmitRpathPatchScript-embedded RPATH chain includes either
sibling from-source mirrors OR nix-store-stub deps will resolve
cleanly because (a) sibling mirrors are preserved at the same
absolute paths via Phase 1, (b) nix-store closure is computed and
mirrored via Phase 2-3.

EVIDENCE-COMMAND TRANSCRIPT
---------------------------
Reproduce locally (eli-wsl, post-M9.R.25.5.1):

  cd /opt/repro/reprobuild/recipes/reproos-iso
  rm -rf /tmp/m9r25_stage
  REPRO_BASE_ROOTFS_DISABLE=1 SOURCE_DATE_EPOCH=1735689600 \
    nix-shell -p patchelf --run \
      "bash scripts/stage-de-rootfs.sh /tmp/m9r25_stage"

  STAGE=/tmp/m9r25_stage
  LDLINUX=$(find $STAGE/nix/store -maxdepth 3 \
              -name 'ld-linux-x86-64.so.2' 2>/dev/null | head -1)
  LD_PATH=""
  while IFS= read -r line; do
    [ -z "$line" ] && continue
    d="$STAGE$line"
    [ -d "$d" ] && LD_PATH="$LD_PATH:$d"
  done < $STAGE/etc/ld.so.conf.d/zz-reproos-overlay.conf

  for binname in sway kwin_wayland mutter sddm sddm-greeter-qt6 \
                 plasmashell reproos-installer repro; do
    BIN=$(readlink -f $STAGE/usr/bin/$binname)
    LD_LIBRARY_PATH="${LD_PATH#:}" "$LDLINUX" --list "$BIN" \
      2>&1 | grep -cE 'not found|error|cannot'
  done
  # Expected: 0 0 0 0 0 0 0 0
