M9.R.53 — reproos-image nix-shell -p leak closed via runtimeDeps declaration
============================================================================

Task
----
Close the third from-source discipline leak in the ReproOS build chain.
Previous leaks: M9.R.25 (apt-fallback for DE dependencies) and M9.R.46
(/nix/store bleed-through in the staged rootfs).  This one: operators
had to wrap `repro build recipes/reproos-image` in
`nix-shell -p qemu grub2 parted gptfdisk dosfstools e2fsprogs rsync
util-linux kmod` because the recipe declared no runtime tool set --
the shell action carried an empty toolIdentityRefs slot, the M9.N
Batch B tool-identity resolver skipped PATH plumbing entirely, and
missing tools surfaced only at run time as `command not found` from
inside the shell script.

provisionable by a store-managed catalog).  Pinned to an absolute
path via a canonical-locations probe (M9.R.53.2 + M9.R.53.5).

Phase B — recipe update
-----------------------
Recipe now declares the full runtime tool set via `runtimeDeps:` +
captures the shell action's returned BuildActionDef and calls
`appendRegisteredActionToolIdentityRefs(action.id,
reproosImageRuntimeTools)` to wire the refs into the M9.N Batch B
resolver.

M9.R.53.3 widens the DSL-port M9.R.5a convention bridge in
`libs/repro_project_dsl/src/repro_project_dsl/macros_a.nim` to union
`pkg.runtimeDeps` into `pkg.toolUses` (previously only `pkg.toolUses`
and `pkg.nativeBuildDeps` were folded), so recipes that declare
tools in the semantically-correct slot (`runtimeDeps:`) don't need
to duplicate them into `uses:` for the resolver to see them.  All
existing from-source recipes carry `runtimeDeps: discard` (empty
seq) so the union is byte-identical to pre-M9.R.53 for them.

M9.R.53.4 collapses the M9.R.53.1 twin declaration to a single
semantic slot: `uses:` retains only `sh` + `bash` (the DSL
primitives the `shell()` typed-tool call depends on); the full 35-
tool operational set lives in `runtimeDeps:`.

Phase C — sudo fix
------------------
M9.R.53.2 + M9.R.53.5: introduce `SUDO` shell variable, probe
canonical host locations (/usr/bin/sudo, /run/wrappers/bin/sudo,
/usr/local/bin/sudo) for a setuid-or-executable match, and rewrite
every `sudo <cmd>` invocation (22 sites) to `"$SUDO" <cmd>`.
Removes `sudo` from the required-host-tool loop (validated
separately as the escape hatch, not through the resolver).

Phase F — honest remaining gaps
-------------------------------
1. Dev-shell provisioning of qemu / grub2_efi / parted / etc. is
   NOT automated by env.ps1 -- callers still have to enrich the
   outer PATH somehow (dev-shell nix-shell -p list, NixOS system
   packages, apt-installs on Debian, etc.).  M9.R.53 closes only
   the RECIPE-side gap (declare + fingerprint the tool set); the
   dev-shell-side gap (auto-provision the tools when the operator
   enters the reprobuild dev shell) is a smaller downstream
   milestone.  This is a much narrower gap than the "nix-shell -p
   wrap around repro build" leak M9.R.53 closes -- the operator
   now sees a structured "tool-resolution failed: <name> requested
   by uses ..." diagnostic naming each missing tool BEFORE any
   shell action fires.

2. QEMU boot smoke (M9.R.50-shape) stalls on systemd's emergency-
   shell root-password prompt when the M9.R.51 boot-from-disk
   initramfs successfully pivots to the rootfs but the rootfs has
   no configured root password (repro build uses the user's
   password_hash for the non-root user only, per auto-config-
   minimal.toml).  The Phase E host-side DE smoke is a stronger
   signal than the interactive rescue shell probes M9.R.50 tried
   -- the binaries are provably invocable.  A follow-up milestone
   should either (a) inject a root password stanza into the shadow
   emit path, (b) mark the emergency shell for autologin via
   systemd.default_target=multi-user.target on the kernel cmdline,
   or (c) drive the DE smoke via `guestmount` / libguestfs against
   the qcow2 without booting.

3. The M9.R.53 fold (`pkg.runtimeDeps` -> `toolUses`) does not
   backpropagate to the individual `executable`-level build:
   blocks that autotools/cmake/meson conventions emit.  For a
   from-source recipe whose `build:` block delegates to
   `autotools_package(...)` etc., `runtimeDeps` still doesn't
   thread into the constructor's per-action `toolIdentityRefs`
   because those constructors call `appendRegisteredAction
   ToolIdentityRefs(action.id, ...)` with only `buildDeps +
   nativeBuildDeps` (M9.R.14e.5).  Widening the constructor fold
   is a separate milestone -- outside M9.R.53's scope, and
   irrelevant to the reproos-image recipe which uses a plain
   `shell()` action rather than a from-source constructor.
