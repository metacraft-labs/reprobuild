EXECUTIVE SUMMARY
=================

M9.R.25 spec said all content-addressed paths live at
``/repro/store/<hash>-<name>/``.  The pre-fix ISO carried a 1.32 GiB
/nix/store closure with 106 unique prefixes referenced by 1143
from-source ELFs.  This file documents the M9.R.46 milestone:

  * M9.R.46.1 measured the leak baseline.
  * M9.R.46.2 implemented a stage-time relocate that moves every
    /nix/store/<hash>-<pkg>/ to /repro/store/<hash>-<pkg>/ at the
    SAME hash + rewrites every ELF's RPATH + PT_INTERP + every
    cross-prefix symlink target + every #!/nix/store shebang.
  * M9.R.46.4 closed two correctness bugs (cp -a-preserved 555
    perms blocking mv + patchelf).
  * M9.R.46.5 measured the post-fix Phase E leak audit at ZERO.
  * M9.R.46.6 added the MINIMAL glibc cache carve-out — 5 symlinks
    under /nix/store/<hash>-glibc-X.Y/etc/ld.so.cache that forward
    to /etc/ld.so.cache — needed because glibc bakes the cache
    path into ld.so's .rodata at compile time, and patchelf
    cannot rewrite .rodata.

The architectural debt is CLOSED at the package-level + ELF-level
+ symlink-level + shebang-level (M9.R.46.1 Phase A baseline shows
106 prefixes / 1143 ELFs / 1.32 GiB; the post-fix audit shows 0
prefixes / 0 ELFs / 0 bytes).

The remaining gap is a NEW investigation surface: Nim's
``{.dynlib: "libclingo.so".}`` pragma is resolved at compile time
by the Nim toolchain walking the linker's search path inside
``nix develop``; the absolute ``/nix/store/<hash>-clingo-5.8.0/lib/
libclingo.so`` path lands in the binary's ``.rodata`` and is passed
verbatim to ``dlopen()`` at runtime.  patchelf does not rewrite
.rodata.  The relocated ISO's /repro/store/<hash>-clingo/lib/
libclingo.so DOES exist (and the m9r28 path /nix/store/<hash>-
clingo/lib/libclingo.so does not), so ``repro hardware probe``
fails Phase 1 with ``could not load: libclingo.so``.  This is a
NEW investigation surface — fixing it requires either rebuilding
the Nim binaries to remove the compile-time path resolution
(``dynlibOverride`` per the clingo_bindings.nim source comment),
OR a second carve-out class (per-Nim-binary content-data-path
forwarding symlinks under /nix/store).

INSTALLER PHASE 1 BLOCKER (handed off)
=======================================

Installer reached Phase 1 + ran ``repro hardware probe``.  The
subprocess failed:

  [2026-06-26 14:28:31] Phase 1: probing hardware...
  [2026-06-26 14:28:31] $ /usr/bin/repro hardware probe ...
  [2026-06-26 14:28:31]   could not load: libclingo.so
  [2026-06-26 14:28:31] hardware probe failed

strace shows the SPECIFIC failure:

  636  1782484111.689633 openat(AT_FDCWD, "/nix/store/kgibywhn...
       -clingo-5.8.0/lib/libclingo.so", O_RDONLY|O_CLOEXEC) = -1 ENOENT

Root cause: ``libs/repro_solver/src/repro_solver/clingo_bindings.nim``
declares ``const clingoLib* = "libclingo.so"`` and uses it via
``{.push cdecl, importc, dynlib: clingoLib.}``.

Nim's documentation + the comment block at lines 51-81 of that
source file say explicitly: ``Nim resolves {.dynlib: const-string.}
at compile time: the Nim compiler walks the linker's search path
and bakes the resolved absolute library path into the generated
C.``  At compile time inside ``nix develop``, the resolved path
was ``/nix/store/kgibywhn-clingo-5.8.0/lib/libclingo.so``; that
absolute string landed in the binary's ``.rodata`` section.

patchelf does NOT rewrite .rodata.  The M9.R.46 relocate cannot
fix this.  The fix shape is one of:

  (a) Rebuild every Nim binary with ``dynlibOverride`` to make
      ``{.dynlib: clingoLib.}`` truly bare-name (``dlopen
      ("libclingo.so")`` → ld.so.cache lookup, which the M9.R.46.6
      carve-out routes correctly).  Per the clingo_bindings.nim
      source comment, an earlier attempt (37e3990) broke every
      provider call site, so this needs careful re-design.

  (b) Add a second carve-out class — per-Nim-binary content-data-
      path forwarding symlinks under /nix/store.  For libclingo
      specifically: ``ln -s /repro/store/<hash>-clingo-5.8.0/
      lib/libclingo.so /nix/store/<hash>-clingo-5.8.0/lib/
      libclingo.so``.  Doable but feels like the wrong shape:
      it whitelists data-path leaks one at a time, and we need
      to audit every binary's strings output.

  (c) Rebuild the Nim binaries with ``--passC:-DCLINGO_LIB_PATH=
      /repro/store/<hash>-clingo-5.8.0/lib/libclingo.so`` so the
      const string is the new location.  Same shape as (a) but
      keeps the eager-load semantics.

(c) is probably the smallest delta given the existing
architecture.  Recommended for the M9.R.47 follow-up milestone.

HONEST REMAINING GAP
====================

M9.R.46's primary scope — close the M9.R.25 architectural debt
that left 1.3 GiB of /nix/store on the live ISO — is CLOSED at
the ELF/symlink/shebang layer.  The post-fix ISO has zero ELFs
referencing /nix/store, no package content under /nix/store, and
only a 5-symlink glibc-cache forwarding stub there.

Two smaller gaps remain:

  GAP 1 (smaller than M9.R.46 task brief envisioned): Nim's
  ``{.dynlib: const-string.}`` pragma bakes /nix/store absolute
  paths into binary .rodata at COMPILE time, before any patchelf
  rewriting opportunity.  The most prominent victim is
  libclingo.so referenced from libs/repro_solver/src/repro_solver/
  clingo_bindings.nim, which blocks Phase 1 hardware probe and
  thus the install-root rsync + DE smoke.  Fix shape (c) — rebuild
  the Nim binaries with -DCLINGO_LIB_PATH=/repro/store/<hash>-
  clingo-5.8.0/lib/libclingo.so — is a one-recipe-rebuild delta;
  M9.R.47 follow-up.

  GAP 2 (separate surface): from-source dbus.service still fails
  on the live ISO.  Pre-M9.R.46 it also failed; M9.R.46 didn't
  worsen this.  Unrelated to /nix/store; the from-source dbus
  needs a runtime config audit.  Hand off to whoever next
  investigates the from-source dbus recipe.

The M9.R.46 milestone closes its OWN gap (the package-level /nix/
store leak) and hands off a SMALLER follow-up gap (Nim-baked
dlopen string forwarding) with full evidence.
