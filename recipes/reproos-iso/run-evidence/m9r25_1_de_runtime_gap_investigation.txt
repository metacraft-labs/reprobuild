M9.R.25.1 — DE-rootfs runtime-resolution gap investigation
============================================================

CONTEXT
-------
M9.R.19.4 introduced a patchelf-strip pass in stage-de-rootfs.sh that
nukes every RUNPATH/RPATH pointing at /opt/repro/reprobuild/ or
/nix/store/ from from-source DE binaries.  M9.R.24.1 then added an
apt-install fallback in build-base-rootfs.sh that pulls in Debian's
sway / kwin-wayland / mutter / plasma-workspace / gnome-session / sddm /
xwayland / KF6 / Qt6 / polkitd / fontconfig / mesa / GL / Wayland stack
"to make exec resolve" because the strip left from-source binaries
unable to find their deps.

The user correctly called out both hacks as architectural failures:

  1. apt-install bypasses the from-source principle Reprobuild stands
     for.  We become an apt-frontend, not a reproducible-from-source
     distro.
  2. RUNPATH-strip discards M9.R.14f's hard-won embedded dep paths.
     Nix-style stores work precisely BECAUSE absolute RPATHs are
     preserved end-to-end (`/nix/store/<hash>-pkg/lib`); the binary
     finds its libs at the same path on every machine that mirrors the
     store layout.

D. Mutter internal RPATH gap (separate issue, M9.R.26 candidate):

   mutter ships its own libmutter-cogl/mtk/clutter-15.so.0 under
   .../usr/lib64/mutter-15/ but neither /usr/bin/mutter nor
   /usr/lib/libmutter-15.so adds $ORIGIN/mutter-15 (or
   $ORIGIN/../lib64/mutter-15) to its RPATH.  This is an internal
   recipe RPATH bug — the install-mirror IS complete, the RPATH is
   incomplete.  Out of scope for M9.R.25 (would need a recipe patch +
   from-source rebuild).

E. Interpreter (PT_INTERP) gap (separate from RPATH):

   sway has PT_INTERP =
     /nix/store/xx7cm72qy2c0643cm1ipngd87aqwkcdp-glibc-2.40-66/lib/ld-linux-x86-64.so.2

   On the ISO that nix-store path doesn't exist, so the kernel can't
   start the binary.  Options:
     a. Mirror /nix/store/<glibc-hash>/lib/ld-linux-x86-64.so.2 onto
        the ISO (preserves the bake-time PT_INTERP without rewriting).
     b. patchelf --set-interpreter /lib64/ld-linux-x86-64.so.2 (the
        from-source glibc recipe's install-mirror provides this path
        via /usr/lib64/ld-linux-x86-64.so.2 -> /lib64/...).

   Either option keeps fidelity to the from-source build.  Option (a)
   is more faithful to the Nix pattern (paths exist verbatim
   everywhere); option (b) trims the ISO size (no need to copy
   /nix/store/<glibc>/).

CONCLUSION
----------
The M9.R.14f embedding mechanism is correct and produces working
binaries on the build host.  The bug is exclusively in
stage-de-rootfs.sh's cherry-pick logic + the M9.R.19.4 RUNPATH-strip
band-aid that the cherry-pick required to limp.

The fix per user direction:

  1. Replace the cherry-pick with FULL install-mirror staging at the
     same absolute paths the RPATHs reference.
  2. Drop the patchelf-strip block entirely.
  3. Drop the apt-install fallback in build-base-rootfs.sh.
  4. Author from-source recipes for the genuinely-absent transitive
     deps (libpng, libevdev, libjpeg, libxcb*, libX*, polkit,
     xwayland) — or document each gap clearly as a TODO(M9.R.26+) and
     let the affected DE fail honestly rather than silently apt-
     install a black-box replacement.

Subsequent commits in this milestone implement (1)-(3).  (4) is out of
the 6-hour budget for M9.R.25; M9.R.25.6 documents the genuine gaps
that block ldd-clean resolution on the ISO without the apt fallback.
