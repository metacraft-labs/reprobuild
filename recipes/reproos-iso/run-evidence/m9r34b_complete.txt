The gap (Phase A — characterised)
---------------------------------

Reprobuild commit 121b3629 ("Fail fast with a diagnostic when RunQuota
stops responding mid-wait") added Windows-breaking call sites that
depended on a runquota client API surface that was claimed to not yet
exist on the runquota sibling:

  1. ``libs/repro_runquota/src/repro_runquota.nim:399`` calls
     ``sessionPtr[].pollNextGrantBounded(boundedReadMs)`` returning
     ``{kind: grantPollTimeout|grantPollFrame, decisions: seq[OfferedLease]}``.

  2. ``tests/unit/t_runquota_grant_wait_unresponsive.nim`` +
     ``t_runquota_grant_wait_wedged_connected.nim`` exercise
     ``pollNextGrantBounded`` and ``client.daemonStatus(timeoutMs = ...)``.

  3. The previous runquota surface had only:
       * ``pollNextGrant`` (no Bounded variant) at line 273.
       * ``daemonStatus`` with NO ``timeoutMs`` parameter at line 419.
       * Unbounded ``readResponse``, ``receiveFrame``, ``readExact``.

Investigation outcome (Phase A — actual state)
----------------------------------------------

Upstream runquota's ``main`` branch ALREADY closed this gap, ahead of
this sub-agent's session, via two commits authored by Zahary +
Claude Opus 4.8:

  3cc9742 (2026-06-18) fix(client): bound the connection handshakes with
                       a timeout
                       --> adds ``timeoutMs = 0`` optional parameter to
                       ``readResponse`` / ``receiveFrame`` / ``readExact``;
                       backend uses ``poll(POLLIN)`` for Unix with
                       ``hasDataBuffered()`` gating to avoid spurious
                       timeouts on the buffered Socket abstraction.

  aa3a301 (2026-06-24) Add bounded grant-stream polling and liveness-
                       probe status reads
                       --> adds ``GrantPollKind`` (``grantPollFrame`` /
                       ``grantPollTimeout``), ``GrantPollFrame`` (with
                       ``decisions: seq[OfferedLease]``),
                       ``pollNextGrantBounded(timeoutMs)`` with single-
                       in-flight GrantNext bookkeeping (parks the
                       request id on the session so a wedged daemon
                       cannot accumulate buffered late responses), and
                       a ``daemonStatus(timeoutMs = 0)`` overload that
                       raises on bounded timeout.

The names + shapes match exactly what reprobuild 121b3629 calls.

A duplicate effort by this sub-agent (commit 69a7c0d, pre-rebase) added
a parallel API surface with similar names but a different internal
design (``BoundedReadOutcome`` enum, separate ``receiveFrameBounded``
proc). That commit was DROPPED via ``git reset --hard`` once the
upstream API was discovered — there was no need to introduce a second
parallel design when the existing upstream one already satisfied the
reprobuild call sites.

Final runquota tree state: clean on origin/main at aa3a301; no local
commits authored by this sub-agent.

The wedged-connected end-to-end test uses Unix-socket primitives
(``unixEndpoint(/tmp/...)``, ``bindEndpoint(unixEndpoint(...))``) and
is Linux-only by construction. Making it cross-platform (named-pipe
equivalent on Windows) is OUT OF SCOPE for M9.R.34b; the bounded I/O
surface upstream uses ``poll(POLLIN)`` on Unix and would need an
overlapped-I/O or PeekNamedPipe equivalent on Windows for a faithful
end-to-end test. The fake-liveness unit suite already covers every
branch of the awaitGrant loop on Windows.

Windows bounded path (runquota commit 10c42a9 — produced by this sub-agent)
---------------------------------------------------------------------------

Upstream aa3a301 added the public ``timeoutMs`` parameter to
``readExact`` / ``receiveFrame`` / ``readResponse`` but the
named-pipe branch ignored it and dispatched straight to the
unbounded ``ReadFile`` (``winReadExact``). So on Windows the
reprobuild call sites compiled cleanly but a wedged daemon that
accepted the named-pipe client would still block the engine forever
— the bounded surface was a NO-OP on Windows.

10c42a9 closes the gap by adding ``winReadExactBounded``: pipe
handles are opened in synchronous mode (no
``FILE_FLAG_OVERLAPPED``), so the bound is enforced by polling
``PeekNamedPipe`` with a yielding ``sleep(1ms)`` until at least one
byte is queued (then ``ReadFile`` cannot block — only short-read),
or the deadline elapses. ``ERROR_BROKEN_PIPE`` /
``ERROR_PIPE_NOT_CONNECTED`` on the peek surface as false
immediately. ``timeoutMs == 0`` still routes to the legacy
unbounded ``winReadExact`` so long-running readers that have not
opted in to bounding keep their semantics.

``tests/unit/t_bounded_pipe_read.nim`` is a Windows-side unit guard
that forks a real silent-stub named-pipe server in the same binary,
has the stub accept the client and stay permanently silent, and
asserts that ``receiveFrame(timeoutMs = 300 / 100)`` on the client
returns false within a small multiple of the timeout. A real-time
guard turns a regression (the Windows branch silently reverting to
``ReadFile``) into a CI failure rather than an infinite hang.
